# Redteam Submission Check

작성일: 2026-05-27

## 1. 식별정보 노출 리스크

- 위험: 제출 화면에서 실제 학교명·주소·전화번호·홈페이지가 노출될 수 있다.
- 현재 대응: `NEXT_PUBLIC_DEMO_ANONYMIZE=true`일 때 학교명은 `A초`, `B중`, `C고` 형식으로 바꾸고 지역은 `서울 A권역`으로 표시한다. 지도는 배경 이미지만 사용하고 마커는 실제 좌표가 아닌 임의 분포로 표시한다.
- 남은 한계: 익명화 false 모드는 내부 검토용이므로 제출 전 환경값 확인이 필요하다.
- 발표 답변 문장: "제출용 화면은 익명화 모드로 운영해 개별 학교 식별정보를 표시하지 않습니다."

## 2. 점수 방향 오해 리스크

- 위험: 점수가 높을수록 좋은 학교로 오해할 수 있다.
- 현재 대응: 홈, 지도, 우선순위, 데이터, 상세 화면에 `점수가 높을수록 지원 필요 신호가 큽니다` 문구를 배치했다.
- 남은 한계: 숫자형 지수는 여전히 비교 인상을 줄 수 있다.
- 발표 답변 문장: "이 점수는 성과 점수가 아니라 공개자료상 지원이 먼저 필요한 정도입니다."

## 3. 학교 서열화 리스크

- 위험: 목록과 지도 TOP 3가 학교 줄 세우기로 보일 수 있다.
- 현재 대응: `우선 지원`, `보완 검토`, `현장 확인 우선`, `다음 행정 조치` 중심으로 표현했다.
- 남은 한계: 지원 자원 배분을 위해 비교는 필요하다.
- 발표 답변 문장: "비교 목적은 우열 판단이 아니라 한정된 지원 자원을 어디에 먼저 투입할지 정하는 것입니다."

## 4. 신뢰도 C 처리 리스크

- 위험: 결측이 많은 학교가 높은 점수로 우선 지원에 섞일 수 있다.
- 현재 대응: 신뢰도 C는 점수와 별도로 `현장 확인 우선`으로 분류하고 지도·목록·상세에서 회색 계열로 표시한다.
- 남은 한계: 현재 데이터에는 신뢰도 C 사례가 없어 조건부 로직으로 검증한다.
- 발표 답변 문장: "자료가 부족한 경우 점수로 밀어붙이지 않고 현장 확인 대상으로 분리합니다."

## 5. 대리지표 한계

- 위험: 공개자료 대리지표가 교실 안 실제 활용을 충분히 설명하지 못할 수 있다.
- 현재 대응: `/data` 화면에서 직접지표, 대리지표, 추가자료 필요 항목을 분리했다.
- 남은 한계: 교원 연수 이력과 AIDT·LMS 로그는 기관 협력이 필요하다.
- 발표 답변 문장: "1단계는 공개자료로 가능한 지원 신호이고, 추가자료가 연결되면 2단계로 정밀화합니다."

## 6. AI 활용 오해 리스크

- 위험: AI가 행정 결정을 대신하는 것으로 오해될 수 있다.
- 현재 대응: 규칙 기반 산식과 AI 요약·분류 보조 역할을 `/data`와 보고서에서 구분했다.
- 남은 한계: 발표에서 AI 활용 위치를 짧게 설명해야 한다.
- 발표 답변 문장: "AI는 산식의 결정을 대체하지 않고 취약 요인 라벨링과 지원 유형 매칭을 보조합니다."

## 7. 시연 동선 리스크

- 위험: 별도 설명 화면이 있으면 실제 대시보드 가치가 늦게 전달된다.
- 현재 대응: 상단 메뉴에서 설명 화면을 제거하고 `/demo`는 홈으로 redirect한다. 시작 화면은 `/map`이다.
- 남은 한계: 발표자는 지도 TOP 3에서 상세 리포트로 들어가는 동선을 연습해야 한다.
- 발표 답변 문장: "격차지도에서 바로 우선 지원 대상과 점수 근거를 보여주고 상세 리포트에서 다음 조치까지 연결합니다."
